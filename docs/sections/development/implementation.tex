\section{Código Fuente}

La \textbf{implementación actual} del proyecto se concentra en la carpeta \texttt{app-tfg/}, mientras que la carpeta \texttt{docs/} almacena la memoria técnica en \LaTeX. Dentro de la aplicación principal se distinguen los siguientes bloques:

\begin{itemize}
    \item Carpeta \textbf{\texttt{app/}}: páginas, \textit{layouts}, \textit{Route Handlers} y componentes de interfaz.
    \item Carpeta \textbf{\texttt{app/api/}}: \textit{endpoints} internos para autenticación, perfil, administración, clientes, catálogo, pedidos, cobros, comunicaciones, auditoría, soporte técnico y operativa comercial.
    \item Carpeta \textbf{\texttt{app/components/}}: componentes reutilizables de interfaz, mapas, formularios, tablas y vistas por rol.
    \item Carpeta \textbf{\texttt{app/hooks/api/}}: \textit{hooks} cliente para encapsular llamadas \textit{HTTP}, estado de carga y mensajes de error.
    \item Carpeta \textbf{\texttt{lib/}}: contratos compartidos, lógica de negocio, integraciones externas, seguridad y acceso a datos.
    \item Carpeta \textbf{\texttt{lib/commercial/}}: planificación diaria de rutas y lógica operativa del módulo comercial.
    \item Carpeta \textbf{\texttt{lib/clients/}}: cálculo reutilizable de \textit{ETA} y lectura de reparto para la zona cliente.
    \item Carpeta \textbf{\texttt{lib/orders/}}: utilidades transversales del dominio de pedidos, entre ellas la generación y normalización de \textit{QR} y etiquetas de reparto.
    \item Carpeta \textbf{\texttt{lib/notifications/}}: soporte de notificaciones internas, correo \textit{SMTP} y entrega \textit{Web Push}.
    \item Carpetas \textbf{\texttt{lib/integrations/}}, \textbf{\texttt{lib/support/}} y \textbf{\texttt{lib/admin/}}: inventarios técnicos, soporte operativo, exportaciones, diagnósticos e informes transversales asociados a \texttt{M7}.
    \item Carpeta \textbf{\texttt{lib/cache/}}: utilidades de \textit{cache} selectiva para lecturas de catálogo y coloración.
    \item Carpeta \textbf{\texttt{lib/security/}}: políticas de \textit{rate limiting} y lógica transversal de seguridad.
    \item Carpeta \textbf{\texttt{lib/typeorm/services/communications/}}: reglas de negocio del centro de comunicaciones, promociones, formaciones, notificaciones y recordatorios.
    \item Carpeta \textbf{\texttt{migrations/typeorm/}}: migraciones versionadas de base de datos.
\end{itemize}

\subsection{Criterios transversales de implementación}

La construcción se ha guiado por varios criterios comunes a todos los módulos, con el objetivo de evitar que cada funcionalidad evolucione de forma aislada:

\begin{itemize}
    \item \textbf{Separación de responsabilidades}: mantener las páginas y componentes centrados en presentación, delegando reglas de negocio en servicios de dominio;
    \item \textbf{Frontera de servidor}: utilizar \textit{Route Handlers} como frontera entre interfaz y servidor, con validaciones de entrada antes de llegar a persistencia;
    \item \textbf{Navegación percibida}: aprovechar convenciones del \textit{App Router}, como los estados \path{loading.tsx}, para mejorar la navegación sin mezclar lógica de negocio con la capa visual;
    \item \textbf{Contratos compartidos}: centralizar contratos y tipos para reducir discrepancias entre cliente y servidor;
    \item \textbf{Trazabilidad de datos}: versionar cualquier cambio estructural de base de datos mediante migraciones trazables;
    \item \textbf{Catálogos controlados}: reutilizar catálogos persistentes para roles, estados y tipos operativos, evitando literales dispersos en el código;
    \item \textbf{Integraciones encapsuladas}: aislar integraciones externas en \textit{helpers} o servicios concretos para facilitar su sustitución o degradación controlada;
    \item \textbf{Coherencia entre roles}: preservar que una misma entidad de negocio pueda ser consultada desde cliente, comercial o administrador sin duplicar reglas incompatibles.
\end{itemize}

Estos criterios permiten que el sistema siga siendo comprensible a medida que crecen los módulos y facilitan la localización de responsabilidades cuando se incorporan nuevas funcionalidades.

\subsection{Reutilización y limpieza estructural}

Durante la fase final de consolidación se ha realizado una revisión específica de duplicidades y organización interna. El objetivo ha sido reducir funciones equivalentes repartidas por componentes y servicios, sin alterar las reglas funcionales ya validadas.

Las \textbf{utilidades transversales} se han agrupado en \path{lib/utils/}. Esta carpeta concentra formateo de fechas, serialización temporal, cálculo de zona horaria de Madrid, parseo y formateo monetario, generación de \texttt{CSV}, normalización de búsqueda y validación básica de textos o enteros. Los servicios de dominio siguen conservando sus errores específicos, pero delegan el comportamiento común en estas utilidades para evitar implementaciones divergentes.

En la \textbf{capa cliente} se ha centralizado el patrón de llamadas \textit{HTTP} en \path{lib/api/client.ts}. Los componentes reutilizan \texttt{requestJson}, \texttt{jsonRequestOptions} y \texttt{getClientErrorMessage} para compartir lectura de respuestas, serialización \texttt{JSON} y extracción de mensajes de error. Del mismo modo, los mensajes de éxito o error de formularios se apoyan en \path{app/components/ui/FeedbackMessage.tsx} y las clases comunes de formularios se agrupan en \path{app/components/ui/form-styles.ts}.

También se han dividido \textbf{pantallas extensas} en piezas más pequeñas. En comunicaciones, el \textit{workspace} administrativo separa formularios, listas, resumen, API cliente y \textit{hook} de orquestación. En pedidos, la carga remota del borrador y las mutaciones principales se han extraído de la vista principal. En el módulo de salón, el detalle se organiza por paneles de ficha, formulario de servicio, historial y plantillas. Esta estructura reduce el acoplamiento entre presentación, estado y reglas de negocio, y facilita que futuras ampliaciones se incorporen sobre puntos de extensión más acotados.

Como ajuste transversal de usabilidad, los modales largos comparten ahora un patrón desplazable para móvil basado en un \textit{overlay} con scroll vertical y panel con altura máxima acotada. Esta decisión evita que formularios extensos de visitas o detalle queden bloqueados en pantallas pequeñas y deja el comportamiento documentado como parte de la consolidación final.

\subsection{Implementación del módulo M2}

El módulo \texttt{M2} se reparte entre las \textbf{rutas de administración de clientes}, la \textbf{zona comercial}, la \textbf{zona cliente} y la lógica de planificación diaria ubicada en \texttt{lib/commercial/}.

Actualmente ya están implementados:

\begin{itemize}
    \item Gestión de clientes profesionales y de sus datos de ficha;
    \item Asignación, desasignación y reasignación de clientes a comerciales;
    \item Geolocalización de establecimientos mediante búsqueda de dirección y confirmación manual sobre mapa;
    \item Configuración operativa del comercial para jornada base, puntos de salida y duración de visitas;
    \item Alta, consulta y seguimiento de visitas comerciales por fecha, tipo y estado, con filtros compactables en el listado para mantener una interfaz móvil manejable;
    \item Finalización explícita de visitas rutinarias desde su detalle cuando no existe flujo de reparto asociado;
    \item Avisos automáticos al cliente cuando se crea una visita o se reubica una visita aplazada, y avisos operativos al comercial cuando el sistema aplaza visitas por horario;
    \item Previsualización de ruta diaria con orden previsto, tiempos estimados, margen disponible e incidencias de franja horaria;
    \item Consulta por parte del cliente de la hora aproximada de reparto cuando existe una visita de entrega planificada para el día.
\end{itemize}

La \textbf{lógica temporal de rutas y estimaciones} no reside en la capa visual, sino que se centraliza en \path{lib/commercial/daily-route-planning.ts}. Esta decisión permite reutilizar el mismo cálculo tanto en el panel comercial como en la tarjeta de \textit{ETA} que visualiza el cliente.

En la revisión final se ha corregido además un desajuste importante entre la \textit{ETA} operativa y la navegación externa. La duración entre paradas ya no depende solo de una velocidad aproximada sobre distancia en línea recta, sino que prioriza tiempos reales por carretera mediante \texttt{OSRM}. Si el servicio no responde, se conserva un \textit{fallback} local por distancia aproximada para no bloquear la operativa. Del mismo modo, el acceso rápido a \texttt{Google Maps} navega hacia la visita actual sin contaminar la \textit{ETA} mostrada con el posible regreso automático al punto de salida al cierre de la jornada.

\subsection{Implementación del módulo M4}

El módulo \texttt{M4} se construye sobre las \textbf{rutas de pedidos}, \textbf{preparación de repartos} y \textbf{cobros} para cliente, comercial y administrador, los servicios de dominio ubicados en \path{lib/typeorm/services/orders/} y la extensión del subsistema de visitas comerciales para soportar repartos reales.

Actualmente ya están implementados:

\begin{itemize}
    \item Creación, recuperación y confirmación de borradores de pedido tanto desde cliente como desde comercial, con buscador por producto, selector de referencia, imagen de producto en cada línea y elección entre entrega por comercial o agencia;
    \item Consulta de historial y detalle de pedidos para cliente, comercial y administrador, con estados de pedido, repartos y cobro diferenciados;
    \item Regla de negocio que limita a un máximo de dos pedidos abiertos pendientes de cobro por cliente;
    \item Página comercial de preparación de repartos, donde un pedido confirmado puede dividirse en varios repartos seleccionando líneas y cantidades concretas;
    \item Registro manual de bultos por reparto, porque el número de paquetes físicos depende de la preparación real del distribuidor y no del número de productos;
    \item Generación de etiqueta \textit{PDF} por reparto: \textit{QR} para entregas comerciales y marca \texttt{AGENCIA} para entregas delegadas en agencia;
    \item Vinculación de repartos comerciales a visitas de tipo \texttt{delivery}, con cierre mediante escáner continuo de cámara en un modal de pantalla completa, \textit{fallback} manual y transición del pedido a \texttt{delivered} solo cuando todos sus repartos están completados;
    \item Liberación automática de los repartos asociados cuando una visita de entrega queda aplazada, para permitir su reasignación posterior sin dejar pedidos bloqueados;
    \item Configuración administrativa del cargo de agencia, aplicada al total del pedido cuando el cliente o el comercial seleccionan esa modalidad de entrega;
    \item Flujo de cobro mediante historial de pagos parciales o totales, con registro de importe, método, fecha, notas y usuario que realiza la acción, y cálculo de saldo pendiente;
    \item Consulta de ETA para cliente solo cuando existe una visita de reparto válida y alineada con un reparto comercial pendiente.
\end{itemize}

La \textbf{lógica de negocio del módulo} se concentra principalmente en \path{order.ts}, \path{order-delivery.ts}, \path{order-settings.ts} y \path{commercial-visit.ts}. El \textit{helper} \path{lib/orders/qr.ts} encapsula la generación y normalización del \textit{QR} de cada reparto, mientras que \path{lib/orders/qr-pdf.ts} genera la etiqueta logística imprimible. \path{lib/clients/delivery-estimate.ts} resuelve la lectura reutilizable de la información de reparto que consume la zona cliente. El módulo \texttt{M6} se apoya en tres bloques principales:

\begin{itemize}
    \item Área administrativa de comunicaciones en \path{/admin/communications};
    \item Páginas de consulta para cliente y comercial en promociones, formaciones y avisos;
    \item \textit{Route Handlers} separados entre administración y consumo por usuario.

La \textbf{lógica de negocio} queda centralizada en \path{lib/typeorm/services/communications/communications.ts}.

Actualmente ya están implementados:

\begin{itemize}
    \item Gestión administrativa de segmentos de cliente, con alta, edición, borrado, código, nombre, descripción y criterios comerciales;
    \item Asignación manual de clientes profesionales a segmentos, con trazabilidad del administrador responsable y notas internas;
    \item Visualización del rango comercial del cliente en \path{/profile} y en la navegación lateral del área cliente mediante un badge metálico, priorizando los segmentos \textit{Platino}, \textit{Oro} y \textit{Plata}, asignando \textit{Plata} como rango base cuando no existe una categoría superior y mostrando las promociones activas disponibles;
    \item Creación, edición, cambio de estado y borrado con confirmación \textit{inline} de promociones con estado \texttt{draft}, \texttt{active} o \texttt{archived}, periodo de vigencia, asociación opcional a producto, línea comercial, cliente o segmento, adjunto \textit{PDF} y miniatura de imagen;
    \item Consulta de promociones activas por parte de clientes y comerciales, aplicando en cliente la visibilidad derivada de su asignación a segmentos y de la jerarquía comercial \textit{Plata}, \textit{Oro} y \textit{Platino};

    \item Inscripción y cancelación de inscripción a formaciones publicadas, validando en servidor la disponibilidad de plazas;

    \item Selección de canales de entrega para promociones y formaciones mediante \texttt{deliveryChannels}, manteniendo el canal interno y permitiendo añadir correo \textit{SMTP} o \textit{Web Push};
    \item Activación y revocación de suscripciones \textit{Web Push} por cliente o comercial desde la pantalla de avisos, persistidas en \texttt{user\_push\_subscriptions};
    \item Envío de correo electrónico mediante \texttt{Nodemailer} cuando existe configuración \textit{SMTP} y envío \textit{Web Push} mediante \textit{VAPID} cuando el usuario tiene suscripciones activas;
    \item Consulta y marcado de lectura de notificaciones por parte de clientes y comerciales, con indicador de avisos sin leer en sus paneles de inicio;
    \item Sincronización de recordatorios internos de visitas del día cuando un cliente consulta sus avisos;
    \item Creación y seguimiento de recordatorios personales con fecha programada y estados \texttt{pending}, \texttt{done} o \texttt{cancelled}.
\end{itemize}

En la \textbf{aplicación de descuentos promocionales} se ha definido una regla de resolución conservadora para evitar acumulaciones ambiguas. Las promociones no se acumulan entre sí: como máximo se aplica una promoción económica por línea de pedido. Para cada producto incluido en el pedido, el servicio busca las promociones activas aplicables al cliente y al propio producto, considerando promociones globales, promociones asignadas al cliente, promociones por rango comercial y promociones vinculadas a producto o línea. En el caso de los rangos, la elegibilidad es jerárquica: las promociones de \textit{Plata} son aplicables también a clientes \textit{Oro} y \textit{Platino}, y las promociones de \textit{Oro} son aplicables también a clientes \textit{Platino}. Entre todas las promociones candidatas se selecciona la que ofrece mayor porcentaje de descuento. Si existe empate, se prioriza la promoción más específica, ponderando de mayor a menor peso la coincidencia por cliente, rango o segmento, producto y línea comercial. Si el empate persiste, se elige la promoción que caduca antes. En las promociones por volumen, el umbral se evalúa contra el subtotal bruto del pedido antes de aplicar descuentos. El porcentaje finalmente aplicado y el total de la línea quedan persistidos como instantánea económica del pedido, de forma que futuras modificaciones de promociones no alteran pedidos ya registrados.

El alcance actual de \texttt{M6} mantiene la \textbf{notificación interna} como fuente principal, pero ya incorpora entrega multicanal opcional por correo electrónico y \textit{Web Push}. Si \textit{SMTP}, \textit{VAPID} o la suscripción \textit{push} no están disponibles, el sistema degrada de forma controlada y conserva la notificación dentro de la aplicación. No se realiza todavía envío por \textit{WhatsApp} ni ejecución autónoma de recordatorios mediante un planificador en segundo plano; estas integraciones quedan documentadas como evolución futura.

\subsection{Implementación del módulo M7}

La aplicación incorpora además varias \textbf{capacidades transversales de \texttt{M7}} que completan la administración, la seguridad técnica y la preparación de integraciones futuras:

\begin{itemize}
    \item Vista de auditoría administrativa y de accesos desde el área \texttt{/admin};
    \item Exportación \texttt{CSV} de accesos y acciones administrativas desde \path{/admin/audit}, protegida por rol de administrador;
    \item Configuración visible de canales de avisos automáticos, cargo de agencia y política de umbrales/recálculo de rangos desde \path{/admin/settings}, con ejecución manual de la reasignación sobre el periodo configurado;
    \item Políticas de \textit{rate limiting} persistidas y diagnosticables desde servicios internos, sin exposición como ajuste ordinario del administrador;
    \item Inventarios técnicos de integraciones y soporte, conservados para scripts, validación y mantenimiento;
    \item Base persistente para integraciones externas, operaciones de intercambio y propuestas de pedido a proveedor;
    \item Estados de carga por segmento para rutas con consultas de servidor relevantes, entre ellas catálogo, coloración, pedidos, fichas de salón, rutas, comunicaciones, auditoría y operaciones;
    \item Optimización \textit{responsive} de imágenes con \texttt{next/image}, especialmente en catálogo, cartas de color y galerías técnicas de servicios;
    \item \textit{Cache} selectiva de lecturas casi estáticas de catálogo y coloración, con invalidación tras escrituras administrativas;
    \item Soporte \textit{PWA} básico mediante manifiesto web, iconos, \textit{service worker} y pantalla \textit{offline} segura;
    \item Compatibilidad mínima de navegador con redirección a pantalla específica de no soporte.
\end{itemize}

El \textbf{inventario de integraciones de soporte} se calcula en \path{lib/integrations/operational-status.ts} y se expone también mediante \path{/api/admin/integrations/status}. En la versión final de esta iteración no se presenta como una página de trabajo del administrador, sino como una lectura técnica que ayuda a validar qué servicios auxiliares sostienen los módulos implementados.

La \textbf{operación empresarial persistente} se centraliza en \path{lib/typeorm/services/admin/enterprise-operations.ts}. Este servicio lista configuraciones, integraciones externas, operaciones recientes y propuestas a proveedor; además permite registrar nuevas operaciones de integración y generar propuestas de reposición desde productos activos del catálogo. La pantalla visible de diagnóstico queda fuera del flujo ordinario, pero los \textit{endpoints} \path{/api/admin/enterprise-operations}, \path{/api/admin/enterprise-operations/operations} y \path{/api/admin/enterprise-operations/supplier-proposals} se conservan como base para futuras integraciones reales.

El \textbf{inventario de soporte técnico} se calcula en \path{lib/support/operational-support.ts} y se expone mediante \path{/api/admin/support/status}. No requiere migraciones ni tablas nuevas, ya que inspecciona activos y configuraciones existentes del propio proyecto: \path{proxy.hosting.ts}, \path{app/manifest.ts}, \path{public/sw.js}, iconos \textit{PWA} y componentes transversales. Su utilidad queda ligada a mantenimiento y pruebas, no a la operativa diaria del negocio.

La \textbf{vista agregada de operación} se calcula en \path{lib/admin/operational-overview.ts} y se expone mediante \path{/api/admin/operations/status}. Este servicio combina contadores de auditoría, estado de políticas de \textit{rate limiting}, resúmenes de los inventarios de integración y soporte, y métricas del registro empresarial persistente para ofrecer una lectura técnica del estado de \texttt{M7}. Como ocurre con las rutas anteriores, queda fuera de la navegación final para no convertir la administración en una documentación viva del sistema.

El \textbf{informe operativo exportable} se genera en \path{lib/admin/operational-report.ts} y se expone mediante \path{/api/admin/operations/report}. Su contenido transforma las secciones y métricas del resumen de operación en un fichero \texttt{CSV}, útil para conservar evidencias de revisión técnica sin introducir una nueva entidad persistente.

La \textbf{exportación de auditoría} se centraliza en \path{lib/admin/audit-export.ts} y se publica mediante \path{/api/admin/audit/export}. El servicio genera bajo demanda ficheros \texttt{CSV} para accesos y acciones administrativas recientes, con límite de filas y ventana temporal configurable por \textit{query string}. De este modo, el administrador puede conservar evidencias sin duplicar información ni añadir nuevas tablas al modelo físico.

El \textbf{diagnóstico de \textit{rate limiting}} se calcula en \path{lib/admin/rate-limit-diagnostics.ts}. Los \textit{endpoints} \path{/api/admin/settings/rate-limits/status} y \path{/api/admin/settings/rate-limits} quedan desactivados por defecto y solo responden si el entorno técnico habilita \texttt{ADMIN\_RATE\_LIMIT\_SETTINGS\_ENABLED=true}. La vista \path{/admin/settings} ya no permite editar estas políticas; se limita a la configuración funcional de avisos automáticos, cargo de agencia y política comercial de rangos.

\section{Migraciones de Base de Datos}

La \textbf{persistencia} se gestiona mediante migraciones de \texttt{TypeORM}. No se han empleado procedimientos almacenados ni disparadores específicos; la lógica de negocio se mantiene en servicios \texttt{TypeScript}, lo que simplifica su trazabilidad junto al resto del código fuente.

Además de las tablas de negocio principales, la base de datos incorpora pequeños \textbf{catálogos normalizados} para roles, estados y tipos operativos. Esta decisión permite que autenticación, auditoría, visitas, rutas, catálogo y pedidos trabajen contra valores controlados y coherentes con las reglas del dominio.
